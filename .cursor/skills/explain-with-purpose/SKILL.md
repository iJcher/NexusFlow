# 目的驱动的步骤说明规范

## 核心原则

每个步骤必须回答三个问题：
1. **为什么**（Why） — 这一步要达成什么目的？解决什么问题？
2. **怎么做**（How） — 具体操作和代码是什么？
3. **注意什么**（What to watch） — 有哪些关键细节、常见陷阱、特殊说明？

## 输出格式规范

### 步骤结构模板

每个步骤按以下结构组织：

```
### 步骤 N：[简明的动作描述]

**目的**：[用 1-2 句话说明这一步解决什么问题、为什么需要做]

[具体操作说明和代码]

> **说明**：[对关键细节、设计决策、参数含义的解释]（仅在必要时添加）
```

### 示例：好的步骤说明

```markdown
### 步骤 1：安装 Prisma 并初始化配置

**目的**：Prisma 是类型安全的 ORM，能根据 schema 自动生成 TypeScript 类型，
避免手写数据库查询和类型定义。初始化后会创建 `prisma/schema.prisma` 文件
作为数据库结构的单一来源。

\`\`\`bash
pnpm add prisma @prisma/client
pnpm exec prisma init
\`\`\`

> **说明**：`prisma init` 会同时生成 `.env` 文件，其中包含 `DATABASE_URL`
> 环境变量。如果项目中已有 `.env` 文件，Prisma 不会覆盖它，而是追加内容。
> 确保 `.env` 已加入 `.gitignore`，防止数据库凭据泄露。
```

### 示例：差的步骤说明（避免）

```markdown
### 步骤 1：安装依赖

运行以下命令：

\`\`\`bash
pnpm add prisma @prisma/client
pnpm exec prisma init
\`\`\`
```

差在哪里：没有说明为什么选择 Prisma、初始化会产生什么效果、有什么需要注意的。

## 分场景的详细要求

### 场景 A：实现新功能

1. 先用一段话概述实现思路和整体架构，让用户有全局视角
2. 每步说明「目的」时，解释这一步在整体架构中的角色
3. 代码示例必须是可运行的完整片段，不能省略关键 import
4. 涉及多种实现方案时，说明选择当前方案的理由

```markdown
<!-- 示例：整体概述 -->
## 实现思路

我们采用「组件 → 组合式函数 → API 层」的分层架构：
- **组件层**负责 UI 渲染和用户交互
- **组合式函数层**封装业务逻辑和状态管理
- **API 层**统一处理网络请求和错误

这样分层的好处是逻辑复用性强，且每层可以独立测试。
```

### 场景 B：排查和修复问题

1. 先定位问题根因，解释「为什么会出这个错」
2. 修复步骤说明「这样改为什么能解决问题」
3. 给出验证方法，让用户能确认问题已修复
4. 如果涉及底层原理，用简洁的方式补充说明

```markdown
<!-- 示例 -->
### 问题根因

`TypeError: Cannot read properties of undefined` 出现在
`user.profile.name` 这一行。原因是 API 返回的数据中 `profile`
字段在用户未完善资料时为 `null`，而代码没有做空值检查。

### 修复方案

**目的**：使用可选链操作符安全访问嵌套属性，避免运行时崩溃。

\`\`\`typescript
// 修复前
const name = user.profile.name

// 修复后
const name = user.profile?.name ?? '未设置'
\`\`\`

> **说明**：`?.` 在 `profile` 为 `null` 或 `undefined` 时短路返回 `undefined`，
> `??` 仅在左侧为 `null`/`undefined` 时使用默认值（不同于 `||`，后者对空字符串和 0 也会触发）。
```

### 场景 C：配置和环境搭建

1. 说明每个配置项的作用，不要只列出配置文件
2. 解释配置值的含义和可选范围
3. 标注哪些是必须修改的、哪些可以保持默认值
4. 提供验证配置是否正确的方法

```markdown
<!-- 示例 -->
### 步骤 2：配置 TypeScript 编译选项

**目的**：为 Node.js + ESM 环境配置正确的编译目标，
确保输出代码能在目标运行时中执行。

\`\`\`jsonc
// tsconfig.json
{
  "compilerOptions": {
    "target": "ES2022",       // Node 18+ 原生支持 ES2022 所有特性
    "module": "NodeNext",     // 使用 Node.js 原生 ESM 模块解析
    "moduleResolution": "NodeNext",
    "strict": true,           // 启用所有严格类型检查
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true       // 生成 .d.ts 声明文件（如需作为库发布）
  }
}
\`\`\`

> **说明**：
> - `target` 设为 `ES2022` 而非更高版本，是因为某些 Node.js LTS 版本尚未完全支持 ES2023+ 特性
> - `module` 和 `moduleResolution` 必须同时设为 `NodeNext`，否则会出现模块解析错误
> - 如果项目不需要作为 npm 包发布，可以去掉 `declaration`
```

### 场景 D：代码重构

1. 说明重构的动机 — 现有代码存在什么问题
2. 展示重构前后的对比，突出改善点
3. 解释重构策略的选择理由
4. 说明如何验证重构没有引入回归

## 代码示例的要求

### 必须包含的内容

- **关键 import 语句**：不要假设用户知道从哪里导入
- **类型标注**：TypeScript 项目中给出完整的类型签名
- **错误处理**：展示异常情况的处理方式
- **实际可用的值**：用贴近真实场景的示例值，而非 `foo`、`bar`

### 代码注释规则

- 只注释「为什么」，不注释「是什么」
- 对非直觉的逻辑、业务规则、性能考量添加注释
- 不要用注释复述代码本身的行为

```typescript
// 差的注释
const timeout = 5000 // 设置超时时间为 5000 毫秒

// 好的注释
const timeout = 5000 // 上游网关 4s 后断开，留 1s buffer
```

## 特殊说明的使用时机

以下场景**必须**添加详细说明：

| 场景 | 说明内容 |
|------|---------|
| 安全相关操作 | 风险点、防护措施、最佳实践 |
| 破坏性操作 | 影响范围、回滚方案、备份建议 |
| 性能敏感代码 | 为什么这样写更快、时间/空间复杂度 |
| 平台/版本差异 | 哪些环境适用、不同版本的行为差异 |
| 反直觉的设计 | 为什么不用"显而易见"的方案 |
| 依赖第三方服务 | 费用、限流、可用性风险 |

## 禁止事项

- 不要只列命令不解释目的
- 不要给出不完整的代码片段（缺少 import、类型等关键部分）
- 不要使用 `foo`、`bar`、`test123` 等无意义占位名
- 不要在不同步骤中对同一概念使用不同术语
- 不要省略错误处理就说"生产环境请自行添加"
